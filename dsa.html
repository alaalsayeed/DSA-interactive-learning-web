🧩 First, make a plan

An algorithm is a clear list of steps. You already use algorithms when you tie shoes, build blocks, or follow a recipe.

🧺 Then, organize things

A data structure is a smart container. The right container makes the job easier and faster.

🚀 Finally, compare speed

Two plans can both work, but one may take far fewer steps. We learn to notice that before the data gets huge.

Level 1 · The big idea

An algorithm is a plan that is clear enough for a robot.

Robots do exactly what you tell them. They do not guess. Build a command list that moves Byte from the blue start square to the golden star.

Build Byte’s instructions

Add one command at a time. A good algorithm has the steps in the right order.

✅ What makes a good algorithm?

Clear: every step has one meaning.

Ordered: steps happen in the right sequence.

Finite: it eventually stops.

Correct: it solves the problem.

🐞 What is debugging?

Debugging means finding and fixing a mistake in the steps. A wrong turn is not a failure—it is a clue! Reset, change one command, and try again.

Superpower: Great programmers are not people who never make mistakes. They are people who learn how to find them.
Level 2 · Smart containers

Different containers are good at different jobs.

A toy shelf, a stack of plates, and a line at a slide all hold things—but their rules are different. Computers use the same idea.

📦 Array: numbered boxes in a row

Every box has an index, or address. Computers usually start counting at 0. Click a box to visit it directly.

Try it: Click any box. You do not need to open the boxes before it.

🥞 Stack: last in, first out

The newest item sits on top, so it leaves first. Think of plates or an undo button.

Level 3 · Put things in order

Bubble Sort makes big numbers “bubble” to the end.

It looks at two neighbors. If they are in the wrong order, it swaps them. The process repeats until everything is sorted.

Level 5 · The growth meter

Big O is a way to talk about how the work grows.

We do not care about one tiny stopwatch reading. We ask: “When the input gets much bigger, how quickly does the number of steps grow?”

How many “tiny jobs” at n = 12?

A kid-friendly speed scale

Do not memorize only the animals. The real skill is spotting the pattern: one job, halving, one pass, repeated passes, every pair, or every possible choice.
Level 6 · Branching data

A tree stores data like a family tree, game menu, or folder system.

In a Binary Search Tree, smaller numbers go left and larger numbers go right. That rule helps us skip whole branches while searching.

🌱 Tree words

Root: the top node.

Parent and child: two connected levels.

Leaf: a node with no children.

Height: how many edges the longest root-to-leaf trip uses.

Tree walk output

“In order” means: visit the left branch, then the node, then the right branch. In a Binary Search Tree, that prints the numbers sorted!

Press “Walk in order.”
Tip: Search compares the target with the current node and chooses only one branch.
Level 7 · Nodes and connections

A graph is a set of places and the roads connecting them.

People in a friendship network, cities and roads, web pages and links, or rooms in a game can all be graphs.

BFS uses a queue

It explores all places one road away, then all places two roads away, and so on. That is why BFS finds the fewest-edge path in an unweighted graph.

Level 8 · Three powerful ideas

Big problems become friendlier when you choose the right strategy.

Explore three patterns you will meet again and again: Divide & Conquer, Greedy choices, and Dynamic Programming.

Split → solve → combine

Instead of sorting one big pile, split it into smaller piles. Tiny piles are easy. Then carefully merge the sorted piles back together.

Examples: Binary Search, Merge Sort, and fast exponentiation all shrink the problem on purpose.

Take the best-looking safe choice now

To make 18 using coins worth 10, 5, and 1, a greedy plan takes the biggest coin that still fits.

Greedy can be wonderfully fast, but it is not always correct. We must prove its choices are safe for the problem.

Save answers so you do not repeat work

Imagine climbing stairs by taking 1 or 2 steps at a time. To count ways to reach stair 6, we can reuse the answers for stairs 4 and 5.

Dynamic Programming works when smaller problems overlap and their answers can be reused.

Rule: ways[i] = ways[i − 1] + ways[i − 2].

🧠 Tiny reflection

When a problem feels hard, ask these four questions:

1. What?What are the inputs and the goal?
2. Which box?What data structure fits the operations?
3. Which plan?Search, sort, split, greedily choose, or remember?
4. How fast?How does the work grow as n grows?
